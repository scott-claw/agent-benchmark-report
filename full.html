Key Findings 11 agents · 33 total runs
  • ✕
    Implementation is a critical failure. 0 of 33 runs resulted in a successfully provisioned API key. Every agent stalled — 31 at email verification, 2 at CAPTCHA. Fixing the signup flow is the single highest-leverage improvement available.
  • ✕
    Tavily wins the choice decision in 24 of 33 runs. Agents explicitly cite "fewer steps to first API call" and "no email verification required" when selecting Tavily. Exa wins on quality — the product is recognized as superior — but loses the conversion on friction alone.
  • !
    Discoverability is inconsistent across agent types. Coding agents (Claude Code, Cursor, Windsurf) surface Exa reliably. General-purpose agents (ChatGPT, Gemini) surface Exa in under 40% of prompts. The gap suggests training data coverage is stronger in coding contexts.
  • !
    No llms.txt or agent-readable documentation detected. Agents fall back to scraping the landing page and GitHub README. Structured agent documentation would improve discoverability and choice scores by an estimated 8–14 points.
  • !
    API design quality is strong — agents who reach the docs rate it highly. This is a meaningful signal: the product itself is not the problem. Agents that get through signup would be successful users. The barrier is entirely upstream.
  • ✓
    OpenAPI spec is well-formed and agent-parseable. Every agent that located the spec was able to construct valid API calls without additional guidance. This is better than most competitors and provides a strong foundation for the discoverability improvements below.

RECOMMENDATIONS
Prioritized Fix List
Ordered by estimated score impact. Each fix is independent — you don't need to do them in sequence.
1
Add an agent-native signup API endpoint
Expose a POST /signup endpoint that accepts email + intended use case and returns a trial API key without requiring email verification. Even a sandbox-scoped key that auto-expires after 24h would unblock all agents. Email verification can be required to upgrade to a production key.
Highest impact Engineering: 1–3 days Affects: Implementation
+28–35 pts
2
Publish llms.txt at exa.ai/llms.txt
A structured, agent-readable file listing your API capabilities, endpoints, authentication requirements, use cases, and pricing. Tavily, Neon, and Vercel all have this. It takes a few hours to write and directly improves how agents describe and choose your product.
High impact Engineering: <4 hours Affects: Discoverability, Choice
+8–12 pts
3
Add /.well-known/anon.json discovery file
Signal to agent frameworks that your service supports programmatic onboarding. Declare your supported auth methods, available scopes, required trust tier, and signup endpoint. This is the standard Anon is building toward — early adopters get preferential placement in the agent directory.
Medium impact Engineering: <2 hours Affects: Discoverability, Implementation
+5–8 pts
4
Remove phone verification from the API key creation flow
Even agents that successfully complete email-based signup (e.g. via OAuth) hit a phone verification gate before they can create an API key. This is a second blocker that affects even partially-capable agents like Devin and OpenClaw. Rate-limit by operator identity instead.
Medium impact Engineering: 1–2 days Affects: Implementation
+4–7 pts
5
Add an MCP server for your API surface
MCP-native integrations are increasingly expected by coding agents (Cursor, Windsurf, Claude Code). An official Exa MCP server would improve both discoverability scores and choice wins in coding contexts, where Exa is already strongest.
Lower urgency Engineering: 1 week Affects: Discoverability, Choice
+3–5 pts
